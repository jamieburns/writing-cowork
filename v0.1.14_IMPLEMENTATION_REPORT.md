## Feature Summary

### Role-Based Task Ownership
- Tasks can be assigned to roles (pm, analysis, review, voice, substance, writer, librarian)
- Custom roles allowed per-project via role_taxonomy.md
- Role combinations supported (e.g., pm+writer, analysis+voice)

### Multi-Mode Kanban Grouping
- **Status grouping** (default): Pending | In-Progress | Done
- **Assignee grouping:** columns per detected role/person
- **Milestone grouping:** columns per phase/stage
- **Flexible column display:** show additional metadata (status, assignee, milestone) on cards

### Filtering & Querying
- `pm-list-tasks --assignee=analysis` — all analysis tasks
- `pm-list-tasks --assignee=voice --status=pending` — pending voice work
- Combined filters support all axes (status, milestone, assignee)

## Decision Log

**Gated Decision #1:** Default kanban grouping for v0.1.14
- ✅ Decided: revert to `status` (was `milestone` in v0.1.8)
- **Rationale:** User feedback shows Status is the primary working lens; milestone view is still available via `--by=milestone`

**Gated Decision #2:** Assignee validation strictness
- ✅ Decided: warn if not in taxonomy, but allow ad-hoc custom roles
- **Rationale:** Flexibility for projects to invent new roles without pre-declaring them; gentle UX via warning

**Gated Decision #3:** Schema column order
- ✅ Decided: Assignee between Milestone and Status
- **Rationale:** Logical grouping—task metadata (ID, description), then context (milestone, assignee), then state (status), then metadata (added, notes)

**Gated Decision #4:** Backward compatibility
- ✅ Decided: parse both 6-col and 7-col schemas; treat missing Assignee as empty
- **Rationale:** Existing Reconciliation project has 6-col todos.md; upgrading should not break existing kanban views
